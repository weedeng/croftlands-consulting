Generative AI for CFOs: What Enterprise LLMs Can Actually Do in a Finance Team

How LLM pipelines using enterprise LLM APIs automate board pack commentary, variance narratives and financial document analysis — with practical examples and the prompt engineering patterns that make them reliable enough to use in production.

The CFO spends an extraordinary amount of time writing text that is largely formulaic: variance commentary for the board pack, narrative descriptions of performance against budget, summaries of financial reports for non-finance stakeholders. These are not low-value activities — the board needs to understand what the numbers mean — but the actual writing is often repetitive: “Revenue was £Xm, ahead of budget by £Yk, driven primarily by stronger-than-expected performance in [segment].” Large language models can draft this text reliably, accurately and in seconds, from structured financial data.

This article is about the practical mechanics of building these pipelines using enterprise LLM APIs — what the inputs are, how the prompts are structured, and what the output quality is like in practice. It is not about using a general-purpose chatbot. It is about building a repeatable, automated pipeline that runs as part of the monthly close process.

The Problem LLMs Actually Solve in Finance

Generative AI does not replace financial judgment. It does not set strategy, challenge assumptions, or tell you whether the revenue growth is real or a result of a change in recognition policy. What it does well is producing first-draft prose from structured data — which is exactly what board commentary is.

The monthly board pack typically contains several narrative sections that follow predictable structures:

  • P&L commentary: Revenue and gross margin versus budget and prior year; key drivers of variances; operating cost movements; EBITDA bridge narrative.
  • Cashflow commentary: Opening and closing cash; key inflows and outflows; movements in working capital; covenant headroom.
  • KPI narrative: Explanation of movements in key metrics — for SaaS, ARR, NRR, churn, CAC payback; for events, forward bookings, yield per delegate, event margin.
  • Risk and outlook: Material risks to the forecast; key assumptions behind the next period outlook.

A CFO who writes these sections manually typically spends 3–5 hours on commentary per board pack cycle. The actual analysis takes 20 minutes; the writing takes the rest. An LLM pipeline can draft all four sections in under 30 seconds from the actuals data. The CFO’s job becomes reviewing and refining the draft rather than producing it from scratch — a much more efficient use of senior finance time.

The Pipeline Architecture

The technical pipeline has three stages: data extraction, prompt construction, and API call.

Stage 1: Data extraction. The actuals data is extracted from the ERP or management reporting system as a structured format — typically a CSV or JSON. For a monthly P&L, this means: actual revenue this month, budget revenue this month, prior year revenue, the same for each major cost line, and a computed variance table. Python (with Pandas) handles this extraction and transformation. The output is a clean, structured Python dictionary containing the numbers the LLM needs.

Stage 2: Prompt construction. The prompt is assembled programmatically by injecting the extracted data into a template. The prompt has three components:

  • A system message that defines the context: “You are a senior CFO writing the finance commentary for the monthly board pack of [Company]. Write in professional finance language, British English. Be direct and specific. Focus on the key variances and their drivers. Do not include generic filler statements.”
  • A data block that injects the structured financial data — the actuals, budget, prior year, and computed variances — in a clean format the model can parse.
  • A task instruction that specifies exactly what to write: “Write the P&L commentary section (200–300 words). Cover: (1) revenue versus budget and PY, key drivers; (2) gross margin movement; (3) operating cost movements; (4) EBITDA bridge.”

Stage 3: API call. The assembled prompt is sent to an enterprise LLM API via its Python SDK. The response is the draft commentary text. The Python script saves it to a file, pastes it into the board pack template, or emails it to the CFO for review — whichever is most useful in the specific workflow.

Prompt Engineering: What Makes It Reliable

The difference between an LLM commentary pipeline that is useful in production and one that produces unreliable output is almost entirely in the prompt engineering. Four patterns make the difference:

Be specific about what “good” looks like. Vague instructions produce generic output. “Write finance commentary” produces boilerplate. “Write 200–250 words of P&L commentary. Open with the revenue figure and the budget variance. State the three key drivers in order of materiality. Close with the EBITDA outturn.” produces structured, useful output that follows a consistent template every month.

Give it the numbers in a clean structure. LLMs are not calculators — they can make arithmetic errors when working from unstructured text. Extract the numbers in Python, compute all variances explicitly, and inject the computed results into the prompt as a structured block. The model’s job is to narrate the variances, not to derive them. This eliminates a whole category of potential errors.

Use few-shot examples for format. If you have a board pack commentary that the board liked — ideally from the prior month — include it in the prompt as an example of the target format and style: “Here is an example of commentary in the correct format and style: [example]. Write this month’s commentary in the same style.” This few-shot approach produces output that matches the specific style and level of formality the board expects, rather than generic finance language.

Chain-of-thought for complex analysis. For commentary that requires reasoning across multiple data points — for example, an EBITDA bridge narrative that needs to reconcile six line items — a chain-of-thought instruction improves reliability: “First list the five largest variances in order of size. Then write the narrative, addressing each in that order.” Asking the model to reason step-by-step before producing output reduces the risk of it skipping material items.

Always review the output. A well-engineered pipeline produces commentary that is correct and useful the vast majority of the time — but it still needs CFO review before it goes to a board. The model does not know about the context the numbers don’t capture: the customer who negotiated a side deal, the cost that was re-classified, the one-off that distorts the comparison. Treat the LLM output as a high-quality first draft, not a finished product. The efficiency gain is in going from zero to 80% in 30 seconds, not in eliminating the review.

Other Finance Applications

Board commentary is the clearest use case, but the same pipeline architecture applies to several other finance tasks:

  • Variance narrative for management accounts: The same approach produces department-level commentary from cost centre actuals — useful for business partner reporting where the volume of commentary is high but each section is short.
  • Financial document summarisation: Long contracts, facility agreements, or investor reports can be summarised reliably using a system prompt that focuses the model on specific clauses: “Extract and summarise: (1) the financial covenants, (2) the restricted payment provisions, (3) the events of default.” This works well for due diligence support where volume is high.
  • Investor report drafting: PE portfolio reporting typically follows a consistent template. LLM-assisted drafting can produce the narrative sections from the underlying data, with the GP adding judgment on strategic items.
  • Audit response drafting: Standard audit queries on accounting policy and disclosure often have standard answers. An LLM pipeline that retrieves the relevant policy documentation and drafts the response can significantly reduce the time the finance team spends on audit support.

Choosing a Model: Which One to Use

The leading enterprise LLMs all produce high-quality output for finance commentary tasks. The practical differences that matter are:

  • Instruction-following and structure — for board commentary you want a model that reliably follows a detailed system prompt, holds a consistent format, and maintains one voice across a long document. Prioritise this for internal reporting.
  • Creativity and ecosystem — some models are more expansive in their language, which suits external-facing narrative, and some sit in richer tooling ecosystems, which matters if you are building a more complex product around the pipeline.

For internal finance commentary, prioritise instruction-following and structure. For more varied or creative output — external investor communications, for example — the differences narrow and most enterprise models perform well.

The more important factor is the quality of the prompt. A well-engineered prompt produces excellent output from either model; a poorly structured prompt produces mediocre output from both. Investment in prompt engineering pays off across the board — and the same system prompt, carefully refined once, can run every monthly close cycle for the entire engagement without modification.
